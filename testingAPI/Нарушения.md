Ресурсные пути и действия в URL:
1. Глагол в пути /createWorkout
2. Действия как подресурсы /workouts/{id}/complete, /workouts/{id}/start
3. Излишняя глубина и дублирование сегментов /data/users/{userId}/data/workouts/{workoutId}/data/exercises 
4. Префикс api в пути /api/users/{userId}/workouts/list, хотя он есть в базовом пути
5. Верхний регистр в пути /USERS/{id}/WORKOUTS, хотя в остальной части - нижний.
6. Несколько путей "/user/{userId}/workouts", "/USERS/{id}/WORKOUTS", "/api/users/{userId}/workouts/list", хотя все возвращают одинаковые списки
7. Версионирование путей без вне базового пути
8. Смешение единственного и множественного числа в путях
9. История и аналитика размазаны по нескольким путям /analytics/history, /history/workouts


HTTP-методы и действия:
1. Удаление через POST "/workouts/{id}/delete"
2. Старт процесса через GET "/workouts/{id}/start", хотя модифицирует состояние.
3. Изменение статуса через query PATCH /workouts с параметрами action, workoutId, хотя action лучше передавать через тело
4. Добавление упражнения к тренировке без ресурса связи POST /workouts/exercises с workoutId в теле запроса, но путь зачем-то содержит workouts
5. update, custom вместо допустимых HTTP‑методов

Статус‑коды и ответы:
1. Создание с 200 вместо 201 POST /createWorkout возвращает 200
2. Нестандартные коды: В /exercises указан 100, в /analytics/improvement указан 250
3. Много 200 без тела и схем

Идентификаторы и параметры:
1. Идентификаторы в query, когда нужен путь GET /progress использует userId и progressId в query
2. Неконсистентное именование userId и user_id

Схемы:
1. Отсутствие обязательных полей на создание в некоторых методах
2. Неопределённые схемы ответов для многих операций
